# i18n (next-intl) — multi-language (es / en / zh)

MEDA apps support international teams. Setup uses next-intl WITHOUT locale routing (no `/en` `/zh` in
the URL) — the language is stored in a cookie. Simpler and enough for internal/shared tools.

## What the CLI sets up
- `src/i18n/request.ts` — reads the `locale` cookie, loads `messages/<locale>.json`.
- `src/messages/es.json`, `en.json`, `zh.json` — translation files (add your keys here).
- `src/features/i18n/language-switcher.tsx` — UI to switch language (sets the cookie + refreshes).
- `next.config.ts` — wrapped with the next-intl plugin.

## One manual step: wrap the app in the provider
In `src/app/layout.tsx`, wrap children with the provider so Client Components can translate:

```tsx
import { NextIntlClientProvider } from "next-intl";
import { getLocale, getMessages } from "next-intl/server";

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const locale = await getLocale();
  const messages = await getMessages();
  return (
    <html lang={locale} suppressHydrationWarning>
      <body>
        <NextIntlClientProvider locale={locale} messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

## Using translations
```tsx
// Client or Server Component
import { useTranslations } from "next-intl";
function Title() {
  const t = useTranslations("transactions");
  return <h1>{t("title")}</h1>;   // es: "Transacciones" · en: "Transactions" · zh: "交易"
}
```

## Switching language
Drop `<LanguageSwitcher />` in your header. It sets the `locale` cookie and refreshes.

## Adding a language
1. Add `src/messages/<code>.json` with the same keys.
2. Add the code to `SUPPORTED` in `src/i18n/request.ts` and to the switcher list.

## Rules (standard)
- All user-facing strings go through `useTranslations`, never hardcoded — so any dev (es/en/zh) reads
  the same keys and the UI works in every language.
- Keep keys grouped by domain (common, nav, transactions...). Same keys across all locale files.
- Money/dates: use `useFormatter` from next-intl OR the MEDA `formatCurrency`/`formatDate` (es-MX);
  be consistent per project.

## How to add translations (the day-to-day workflow for devs)

This is what you do every time you build a feature that shows text to the user.

### 1. Never hardcode user-facing text
```tsx
// ✗ WRONG — hardcoded, only works in one language
<h1>Transacciones</h1>

// ✓ RIGHT — goes through a translation key
import { useTranslations } from "next-intl";
function Title() {
  const t = useTranslations("transactions");
  return <h1>{t("title")}</h1>;
}
```

### 2. Add the key to ALL three JSON files (same structure)
The files live in `src/messages/`. Keep the SAME keys in every file — only the values change.

`src/messages/es.json`
```json
{
  "transactions": {
    "title": "Transacciones",
    "empty": "No hay transacciones"
  }
}
```
`src/messages/en.json`
```json
{
  "transactions": {
    "title": "Transactions",
    "empty": "No transactions"
  }
}
```
`src/messages/zh.json`
```json
{
  "transactions": {
    "title": "交易",
    "empty": "暂无交易"
  }
}
```

### 3. Organize keys by domain (namespaces)
Group keys by feature so they're easy to find and `useTranslations("namespace")` stays scoped:
```json
{
  "common":       { "save": "Guardar", "cancel": "Cancelar", "retry": "Reintentar" },
  "nav":          { "home": "Inicio", "transactions": "Transacciones" },
  "transactions": { "title": "Transacciones", "amount": "Monto" },
  "transfer":     { "title": "Transferencia", "confirm": "Confirmar envío" }
}
```
Then: `const t = useTranslations("transfer"); t("confirm")`.

### 4. Values with variables (interpolation)
```json
{ "greeting": "Hola, {name}", "count": "Tienes {n} movimientos" }
```
```tsx
t("greeting", { name: user.firstName })
t("count", { n: movements.length })
```

### 5. Plurals (next-intl uses ICU syntax)
```json
{ "items": "{count, plural, =0 {Sin movimientos} one {# movimiento} other {# movimientos}}" }
```
```tsx
t("items", { count: n })
```

### 6. Formatting money and dates per locale
For numbers/dates, prefer next-intl's formatter OR the MEDA helpers — be consistent per project:
```tsx
import { useFormatter } from "next-intl";
const format = useFormatter();
format.number(1234.5, { style: "currency", currency: "MXN" });   // es-MX: $1,234.50
format.dateTime(date, { dateStyle: "medium" });
```

### 7. In Server Components
Use `getTranslations` (async) instead of the hook:
```tsx
import { getTranslations } from "next-intl/server";
export default async function Page() {
  const t = await getTranslations("transactions");
  return <h1>{t("title")}</h1>;
}
```

## Checklist for every new feature
- [ ] No hardcoded strings — all through `t("key")`.
- [ ] Key added to es.json, en.json AND zh.json (same key, translated value).
- [ ] Key grouped under the right domain namespace.
- [ ] Variables use `{name}` interpolation, not string concatenation.
- [ ] Money/dates use the formatter (locale-aware), not manual formatting.

## Adding a new language
1. Create `src/messages/<code>.json` with the same keys as the others.
2. Add the code to `SUPPORTED` in `src/i18n/request.ts` and to the `MESSAGES` map.
3. Add it to the `LOCALES` list in `src/features/i18n/language-switcher.tsx`.
